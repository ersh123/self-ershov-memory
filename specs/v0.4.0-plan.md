# Hermes Ershov v0.4.0 — Implementation Plan

## Headline

> *The trust loop and the friction-killer.*

v0.4.0 makes Ershov safe to use in anger (revert, dry-run, selective apply) and removes the harvest-to-create two-step (one command, real sessions, redacted).

## Goals (in priority order)

1. **Operator can undo an apply.** `ershov revert` restores a live root from the artifact's recorded backups and rolls proposal state back from `applied` to a terminal "reverted" state. This is the trust headline.
2. **Operator can preview an apply.** `ershov apply --dry-run` validates + simulates the apply path without writing live state or producing a destructive backup.
3. **Operator can apply only a subset.** `ershov apply --priority {low,normal,high}` and `ershov apply --target-kind {memory,user,skill,fact}` filter which approved proposals land.
4. **`reject --reason` is enforced.** Move the `required=True` flag from argparse into a hard validation in `reject_artifact` so any code path (not just CLI) cannot reject without a reason. Already enforced at the CLI layer; tighten in the command layer too.
5. **Inbox shows what is ready to apply.** `ershov inbox --apply-ready` filters to artifacts where every proposal is either approved or already applied (i.e. nothing blocking apply). Output also feeds the inbox digest.
6. **`create --from-sessions` works.** `ershov create --from-sessions N` and `--from-since 7d` auto-harvest from the local SessionDB and feed the staged create in one step. `--recent` already exists; the new flag renames + adds time-window support + always prints redaction stats.
7. **Discovery is less rough.** `ershov providers list` introspects which providers are available; `--no-llm` is a shorthand for `--provider offline-marker`.

## Non-goals (explicitly out of scope for v0.4.0)

- Cross-artifact dedupe / merging (C-direction) — v0.5.0 candidate.
- Multi-artifact report card — v0.5.0 candidate.
- Skill-as-special-target policy (E-direction) — needs its own design pass; v0.5.0+ candidate.
- New provider families beyond the current built-ins (`offline-marker`, `openai-compatible`, `deepseek`, `ollama`) — only `providers list` discovery.
- Background ershov / auto-consolidation / sleep memory — already in upstream scope, do not duplicate.
- User-facing `/ershov` reflection mode — already #5533 upstream.
- v0.4.0 does **not** include a tag, GitHub release, or PyPI publish. That requires Niko's explicit release gate after this plan finishes.

## Command surface (final)

### New commands

```
ershov revert <artifact>
  [--live-root PATH]            default: artifact.workspace_root
  [--backup-root PATH]          default: <live-root>/.ershov/backups
  [--yes]                       skip the confirmation prompt
ershov apply --dry-run <artifact>
  [--live-root PATH]
  [--backup-root PATH]
  [--priority low,normal,high]
  [--target-kind memory,user,skill,fact]
ershov providers list
ershov create --from-sessions N
  [--from-since 7d]             alternative to --from-sessions
  [--recent N]                  alias for --from-sessions, kept for back-compat
  # all other existing create args unchanged
```

### Modified commands

```
ershov apply
  + --dry-run
  + --priority {low,normal,high}            single value or comma-list
  + --target-kind {memory,user,skill,fact}  single value or comma-list

ershov inbox
  + --apply-ready                            filter to apply-ready rows

ershov create
  # --from-sessions / --from-since / --recent are already there; new
  # behavior: always print redaction count + session count from
  # the harvest step before staging.
```

### Unchanged commands

`summarize, approve, reject, diff, validate, discard, compact, install-cron, digest, status, report-card, update, review, harvest`

## Revert — design specifics

State to track (additive, on `DreamArtifact`):

```python
reverted_at: str | None = None
revert_audit_events: list[dict] = field(default_factory=list)
```

CLI behavior:

1. `load_artifact(artifact_dir)` — must have `status == "applied"`. Reject anything else with a clear error.
2. `revert_artifact(artifact_dir, live_root, backup_root, *, yes=False)`:
   - Resolves backup files from `artifact.backup_paths`.
   - For each backup path, restores the corresponding live file to the pre-apply state.
   - On restore failure: aborts, leaves live state in place, writes an audit event.
   - Marks the artifact `status = "reverted"`, `reverted_at = now_iso()`.
   - Rolls each `applied` proposal back to `approved` (state-wise) and writes an audit event for each.
   - Writes a `REVERT.md` next to the artifact summarizing what was restored and what failed.
3. Confirmation: `revert` prints a one-screen summary of what will be restored and asks for confirmation unless `--yes` is passed. Non-interactive (cron / pipe) requires `--yes`.

Failure modes the implementation must handle:

- Backup file is missing on disk (was deleted / rotated). Fail loud.
- Backup file is corrupt. Fail loud.
- Live file has been modified after apply (drift detection via `text_sha256` of pre-write content if available, or by comparing current file size + sha to the snapshot stored at apply time).
- A proposal was applied but is not in `backup_paths` (shouldn't happen, but guard). Fail loud with the proposal id.

Tests for revert:

- Apply then revert roundtrip: live state returns to pre-apply content.
- Revert on a non-applied artifact raises.
- Revert with a missing backup file raises and leaves live state untouched.
- Revert writes `REVERT.md` and updates `manifest.json` and `audit.jsonl`.
- Drift detection: if the live file changed after apply, revert still restores from backup but records a `drift_detected` audit event.

## Apply — design specifics

`--dry-run`:

- Calls `apply_artifact` with `dry_run=True` flag (new kwarg) that:
  - Runs the same validation pipeline as real apply.
  - Skips backup creation.
  - Does not write to live roots.
  - Returns a structured "what would have been written" report.
- Output prints the per-proposal diff that *would* land, plus a "no live writes performed" footer.

`--priority`, `--target-kind`:

- Filter the proposals that the apply step considers approved.
- The artifact's `applied_proposal_ids` and `backup_paths` reflect only the proposals that actually landed.
- A proposal filtered out stays `approved` (so a later apply with a different filter can still land it).

## Reject — design specifics

Currently `cli.py` line 103 has `required=True` for `--reason`. Promote the constraint down into `commands/review.py:reject_artifact()` so any caller (CLI, library, plugin) must pass a reason. `ReviewError("reason required")` on missing/empty reason.

## Inbox — design specifics

`--apply-ready` filter:

- An artifact row is "apply-ready" when:
  - `artifact.status in {"staged", "approved", "applied"}` (i.e. not `invalid`, not `rejected`, not `discarded`, not `reverted`)
  - All non-rejected proposals are `approved` (or `applied`).
- Filter is composable with `--state` and `--priority`.
- Inbox digest auto-includes a "Ready to apply" section that uses the same filter.

## From-sessions — design specifics

`--from-sessions N` is the new preferred name for `--recent N`. Both work, `--recent` becomes an alias. The behavior is:

1. Call `harvest_recent(recent=N, output_path=<artifact-root>/_sources/recent-sessions.md)`.
2. Print to stdout *before* staging: `harvest: <path>`, `sessions: <count>`, `redactions: <count>`.
3. Pass the harvested bundle as a `--source` to the existing create flow.

`--from-since 7d` parses the time window (suffix `h` for hours, `d` for days, `w` for weeks) and converts it to "most recent N sessions within that window." If the window is set, N is computed from `session_reader.list_recent()` and capped at a sane default (e.g. 50).

Both flags require an explicit `--live-root` to resolve the SessionDB location (fall back to `~/.hermes/state.db`).

## Providers list — design specifics

`ershov providers list` prints a table:

```
NAME                KIND                STATUS    NOTES
offline-marker      offline             always    no API key required
openai-compatible   openai_compat       optional  requires [llm] extra
deepseek            openai_compat       optional  requires [llm] extra and DEEPSEEK_API_KEY
ollama              ollama              optional  requires local Ollama server
```

`STATUS = always|optional|missing` based on whether the provider's optional dep / external service is reachable. For v0.4.0, "optional" is reported if the import succeeds; "missing" otherwise. We do not ping external services in this command.

`--no-llm` shorthand:

- Equivalent to `--provider offline-marker`. The CLI accepts it on `create`, `review`, and `nightly` and translates internally.
